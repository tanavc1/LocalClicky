## Findings & decision (Phase 0)

Direct raw probes of Moondream on the same screenshot:

- **Describe** ("describe what is on this screen") → accurate and concise:
  *"a computer screen displaying a web browser window with a tab open to a website
  called www.w3schools.com."* Moondream is **strong at description / VQA**.
- **Point** ("where should I click… append [POINT:x,y]") → **empty output**. Moondream
  does **not** emit the structured pointing tag. The `750016 tok/s` / `total < TTFT`
  artifact in the table above is exactly this: near-empty responses on the pointing
  prompts.

`qwen2.5vl:3b` is the only installed model that reliably grounds (returns an in-bounds
`[POINT:x,y]`). `qwen3-vl:8b` is far too slow on 16 GB (≈10 s/answer, swapping) and also
failed to point in this run.

**Decision (matches the locked plan):** Moondream 2 becomes the default vision model for
*describing/answering/joking/give-text about the screen*; **pointing turns auto-route to
`qwen2.5vl:3b`** (the grounding model). The fallback is **required**, not optional, because
Moondream cannot point at all. Autotune (Phase 4) decides residency: on 16 GB the resident
pair is {small text model, Moondream}, with the grounding model loaded on demand for
pointing turns (or kept resident on machines with more RAM).

> Note: point-parse/in-bounds here are an honest *reproducibility* proxy (does it return a
> parseable, in-bounds coordinate), not hand-labeled target accuracy. The "after" run in
> Phase 7 uses the same harness + image so the deltas are comparable.
